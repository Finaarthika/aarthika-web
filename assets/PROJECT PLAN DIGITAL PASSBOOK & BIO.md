PROJECT PLAN: "DIGITAL PASSBOOK & BIOMETRIC ACCOUNT LEDGER"
📑 PHASE 1: Data Architecture Design (Account Opening)
Before you write any code, you need a structured "bucket" to collect and hold information during the account opening process. Your database must support text records alongside specialized files for images and biometric markers.
1. Account Opening Registration Form (The Inputs)
When a new customer arrives, you will collect the following immutable fields:
• Unique System ID: An automatically generated registration number (e.g., ACC-2026-0001).
• Full Name: The primary text index (e.g., "Aman Kumar").
• Father's Name: Crucial for local deduplication (e.g., "Suresh Kumar").
• Village/Address: Categorized by local neighborhoods or landmarks.
• Phone Number: A 10-digit mobile number contact vector.
• Profile Photo Capture: A clear, front-facing webcam or camera snapshot stored directly on your server directory.
• Face Matrix Vector: A string array containing 512 numerical data coordinates generated by passing the profile photo through the facenet_512 model.
2. The Multi-Row Grid Search Interface Layout
If you type "Aman" into the search bar, the UI will suppress the ledger buttons entirely and dynamically draw a multi-row grid system containing exactly what you specified:

🔍 PHASE 2: The Dual-Track Search Engine Pipeline
Your system must handle searches via two independent routes: standard keyboard text queries and mathematical face scans.

Track A: Text Query Deduplication Architecture
• The system executes an ongoing wildcard database lookup (WHERE customer_name LIKE '%Aman%').
• If multiple records emerge, it arrays them sequentially based on their Father's Name and Village context.
• The high-resolution profile picture displays as a functional visual anchor on the card layout, allowing you to instantly identify the physical person standing in front of you.
Track B: The facenet_512 Biometric Search Pipeline
You do not need an internet-connected, complex cloud API for this. You can run the facial verification pipeline completely locally on a standard laptop offline:
1. The Camera Step: When the customer walks up, you click a "Scan Face" button on your website. Your site opens your laptop's built-in webcam and snaps a fast image.
2. The Extraction Step: The local system passes that raw image into the facenet_512 model in the background. The model analyzes features (distance between eyes, jaw structure, nose length) and converts the physical face into an immutable array of exactly 512 floating-point decimal numbers.
3. The Similarity Comparison Step: Your database scans the 512-number vector of the person standing in front of you against all the vectors saved during the account opening phase. It calculates the geometric distance between the numbers.
4. The Instant Match: The record with the absolute closest mathematical match is immediately pushed to the top of your screen as a 99% confident match. You click it to confirm.

🗺️ PHASE 4: The Lean Development Roadmap (Phased Approach)
To keep this manageable for your first tech development side quest, tackle the system piece-by-piece rather than building everything at once.
• Milestone 1 (The Storage Layer): Write your localized text database files. Practice creating a dummy user dashboard containing names, fathers' names, villages, phone numbers, and net balances. Ensure you can manually append records.
• Milestone 2 (The Multi-Row Grid View): Build a simple web layout containing a search field. Practice typing a name and making sure the system correctly splits identical names into a multi-row layout showing text descriptions side-by-side with an image placeholder.
• Milestone 3 (The Ledger Actions): Connect your clean Deposit and Withdrawal workflows directly behind that selection link. Ensure that clicking a specific row opens that account's ledger view, blocks overdraft entries, and logs transaction history logs.
• Milestone 4 (The Biometric Layer Upgrade): Integrate the offline Python facenet_512 model libraries. Set up your camera to translate local profile photos into biometric vectors, and pipe those metrics directly into your database search parameters.
